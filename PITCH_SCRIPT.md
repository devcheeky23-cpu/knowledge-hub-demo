## [0:00–1:00]  Slide 2 — Problem

> "ลองนึกถึง developer ในทีมเรา ทุกสัปดาห์เขาต้องถามคำถามเดิมซ้ำๆ — API ตัวนี้รับ field อะไร, ตรงนี้ design ไว้ยังไง, ทำไม flow เป็นแบบนี้ คำตอบมีอยู่จริง แต่กระจัดกระจาย — อยู่ใน PRD บ้าง, ใน chat บ้าง, อยู่ในหัวคนบ้าง"

> "ปัญหาคือ — คนถามก็เสียเวลา คนตอบก็เสียเวลา ความรู้ส่งต่อไม่ครบจาก PM/BA ไปหา dev จาก backend ไป frontend และพอคนเก่งลาออก ความรู้ก็หายไปกับเขาด้วย"

**"สรุปคือ เราไม่มีศูนย์กลางความรู้ และมันกินเวลาทีมทุกสัปดาห์"**

## [1:00–1:30]  Slide 3 — Value

> "สิ่งที่เราเสนอคือศูนย์กลางเดียว ที่ทุกคนถามด้วยภาษาธรรมชาติ แล้วได้คำตอบจากเอกสารจริงของบริษัท"

> "ผลที่วัดได้: dev หาคำตอบเองได้ ไม่ต้องรบกวนใคร, คนใหม่ onboard เร็วขึ้น, ความรู้ไม่หายตอนคนลาออก และระบบยังบอกเราได้ด้วยว่าเอกสารส่วนไหนยังขาด"

**"เราขายผลลัพธ์ ไม่ใช่เครื่องมือ — ประหยัดเวลาทีม และรักษาความรู้ไว้กับองค์กร"**

## [1:30–2:30]  Slide 4 — Approach

> "เบื้องหลังใช้เทคนิคชื่อ RAG หลักการง่ายมาก: ก่อนตอบ ระบบจะไปค้นเอกสารที่เกี่ยวข้องมาก่อน แล้วให้ AI ตอบ*จากเอกสารนั้นเท่านั้น* ไม่ใช่เดาจากความรู้ทั่วไป"

> "สามขอบเขตสำคัญ — หนึ่ง ตอบจากเอกสารที่เรา import เข้าไปเท่านั้น สอง AI ถูกจำกัดให้ใช้แค่ context ที่ดึงมา สาม ทุกคำตอบมี citation ให้ตรวจสอบได้ว่ามาจากไฟล์ไหน"

**"จุดสำคัญคือ ทุกคำตอบตรวจสอบได้ — ไม่ใช่กล่องดำ"**

## [2:30–4:30]  Slide 5 — Demo (โชว์ของจริง — ใช้เวลามากสุด)

> "ขอโชว์ของจริงครับ" *(เปิด mockup)*

**Scene 1 — Found** *(พิมพ์: Order API รับ field อะไร)*
> "ถามเรื่องที่มีในเอกสาร — ระบบตอบ พร้อม citation คลิกดูต้นฉบับได้เลยว่ามาจาก api-spec.md หัวข้อ Order Endpoints"

**Scene 2 — Abstain** *(พิมพ์: ระบบรองรับ GDPR มั้ย)*
> "ทีนี้ถามเรื่องที่*ไม่มี*ในเอกสาร — สังเกตว่าระบบไม่เดา มันบอกตรงๆ ว่าไม่พบข้อมูล นี่คือหัวใจ — ระบบที่เดาแบบมั่นใจ อันตรายกว่าระบบที่บอกว่าไม่รู้"

**Scene 3 — Conflict** *(พิมพ์: payment timeout กี่วินาที)*
> "อันนี้น่าสนใจ — เอกสารสองไฟล์เขียนไม่ตรงกัน ไฟล์นึงบอก 30 วิ อีกไฟล์บอก 60 วิ ระบบไม่เลือกข้างให้ มันโชว์ทั้งสองฝั่งพร้อม citation แล้วให้คนตัดสิน — และยังชี้ให้เราเห็นด้วยว่าเอกสารขัดแย้งกันตรงไหน"

**"สามพฤติกรรมนี้ — ตอบพร้อมอ้างอิง, ยอมรับเมื่อไม่รู้, รายงานเมื่อขัดแย้ง — คือสิ่งที่ทำให้เชื่อถือได้"**

## [4:30–5:30]  Slide 6 — Risk & Control

> "เรื่องความเสี่ยง เราคิดมาแล้ว และหลักการคือ — ความเสี่ยงไม่ได้แปลว่าห้ามทำ แต่แปลว่าต้องมี control"

> "Hallucination — คุมด้วยการบังคับตอบจากเอกสาร + บังคับ citation. ข้อมูลรั่ว — MVP ใช้ mock data, production ค่อยย้ายไป self-host. และทุกคำตอบมี citation ให้คนตรวจก่อนเอาไปใช้ ทุก risk มี owner ชัดเจน"

**"เราพูดข้อจำกัดของตัวเองก่อนที่จะถูกถาม — เพราะนั่นคือความน่าเชื่อถือ"**

**"ขอแค่ 1 project กับเอกสารจริง เพื่อ pilot 2 สัปดาห์ แล้ววัดผลด้วยตัวเลขจริง"**

## Q&A Buffer (Slide 8 — เตรียมตอบ ไม่ต้องพูดเอง)

**Q: ต่างจาก ChatGPT ยังไง?**
> ChatGPT เดาได้ ข้อมูลออกนอกองค์กร และคุม behavior ไม่ได้ ของเราตอบจากเอกสารเราเท่านั้น + citation + คุม retrieval/abstain/conflict เองได้ + deploy ใน infra เราเองได้

**Q: Abstain ChatGPT ก็ทำได้ถ้าสั่ง?**
> ใช่ครับ แต่ของเราทำให้เป็น default ที่การันตี + วัดผลได้ + คุม retrieval เองได้ โดยเฉพาะภาษาไทย ซึ่ง ChatGPT file upload ปรับตรงนี้ไม่ได้

**Q: ทำไมไม่ใช้ ChatGPT ไปเลย?**
> Privacy + คุม retrieval ภาษาไทย + conflict mode + integrate เข้า workflow เราเองได้

**Q: ภาษาไทยเป็นจุดขาย แต่เอกสาร demo อังกฤษ?**
> use case จริงคือ คำถามไทย + เอกสารผสมไทย/อังกฤษ — สถานการณ์จริงของทีมไทย เราเลือก embedding ที่ match ข้ามภาษาได้ ซึ่งเป็นเหตุผลที่ไม่ใช้ default ของ tool

**Q: วัดความถูกต้องยังไง?**
> golden question set 15–20 ข้อ เทียบกับ expected answer ที่เตรียมไว้ รวมเคสที่คำตอบที่ถูกคือ "abstain"
